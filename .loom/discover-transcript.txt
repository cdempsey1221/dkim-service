### Helper functions

**`parse_key_type(record: str) -> str`** (`dkim.py:30–36`)  
Iterates whitespace-split tokens, then semicolon-splits each token to find a tag starting with `k=`. Returns the value uppercased, defaulting to `'RSA'` per RFC 6376 §3.3 if no `k=` tag is found.

**`split_key_by_val(key: str) -> dict`** (`dkim.py:20–28`)  
Positional whitespace split: `tokens[0]` = name, `tokens[2]` = `v=DKIM1;...`, `tokens[3]` = key material, `tokens[4]` = TTL. Returns two record strings. No error handling — crashes on wrong token count.

### What "valid DKIM record" means today

The service has no existing endpoint for full DKIM record validation. `parse_key_type` does only tag extraction. RFC 6376 §3.6 defines the DNS TXT record format for DKIM:

- **Required**: `v=DKIM1` as the first tag
- **Required**: `p=` tag containing the Base64-encoded public key (empty `p=` means key is revoked)
- **Optional known tags**: `k=` (key type; default `rsa`), `t=` (flags), `s=` (service), `h=` (hash algorithms), `n=` (notes)
- The record is semicolon-delimited key=value pairs, typically embedded in a DNS TXT string

## Risks and open questions

1. **What constitutes "valid"?** The goal says "actual valid DKIM record" but does not define the scope:
   - *Syntactic validity only* (correct tag format, required `v=DKIM1`, non-empty or empty `p=`)?
   - *Base64 decodability* of the `p=` public key value?
   - *Cryptographic key parsing* (e.g., attempt to load the RSA/Ed25519 key from the decoded bytes)?
   - *Live DNS lookup* to confirm the record actually exists in DNS?
   The design phase must pick a scope; syntactic + Base64 check is likely the right baseline.

2. **`v=DKIM1` must be first tag per RFC 6376 §3.5** — should the validator be strict (reject if `v=` is not the first tag) or lenient (accept if `v=DKIM1` is present anywhere)?

3. **Revoked keys**: RFC 6376 allows `p=` to be empty (empty string) to signal key revocation. Should an empty `p=` be considered "valid" (structurally correct but revoked) or "invalid"? This distinction may matter for the endpoint's intended use.

4. **Input format ambiguity**: Existing code treats `dkim_record` as a full DNS-zone-file-style string (e.g., `"big-email._domainkey.example.com TXT v=DKIM1; k=rsa; p=..."`) with name, type, and RDATA all in one string. Should the new validator accept the same compound format, or just the raw RDATA portion (the semicolon-delimited tag=value string)?

5. **New library vs. inline logic**: No DKIM library is currently in `requirements.txt`. If the design phase chooses cryptographic key parsing, a library like `cryptography` would be needed and the Dockerfile `pip install` step would need updating.

6. **Response shape**: No precedent for a boolean-result endpoint exists. The design phase must decide the response body — e.g., `{"valid": true}` / `{"valid": false, "reason": "..."}` vs. 200/422 status-based.

7. **Mixed test frameworks**: The new test file could use either `unittest` or `pytest`. `test_dkim_key_type.py` (pytest) is the most recent and is the clearest model; the design phase should commit to pytest for consistency with the latest pattern.

8. **Endpoint name**: Candidates: `/dkim_validate`, `/validate_dkim_record`, `/is_valid_dkim`. Convention in this codebase is snake_case route names (e.g., `/split_by_value`, `/dkim_key_type`).
