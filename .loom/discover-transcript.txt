## Risks and open questions

1. **HTTP method**: The goal says "returns 200 if the json contains the dkim record" but doesn't specify `GET` vs `POST`. Given JSON body parsing, `POST` is the natural choice (consistent with `/split_by_value`), but this should be confirmed.

2. **`k=` tag absent → default RSA?** The canonical DKIM test record in this repo has no `k=` tag. RFC 6376 §3.3 defaults to `rsa` when `k=` is omitted. The new endpoint must decide: (a) default to `"rsa"` when absent, (b) return 400/error, or (c) document that it only works on records that explicitly carry `k=`.

3. **Case sensitivity of key type**: RFC 6376 says `k=rsa`; should the response normalise to uppercase `"RSA"` (as the task description implies) or echo the wire value `"rsa"`?

4. **Record format variability**: The existing record string is a flat whitespace-separated line (`name TXT <tags> ttl`). Real DKIM TXT records may have tags in different orders or combined differently (e.g., `v=DKIM1; k=rsa; p=...` as one token or spread). The parser must handle semicolon-separated tags within the body portion rather than relying on fixed token indices.

5. **What constitutes "parseable"?**: The task says return 200 "if … it can be parsed from the payload." This implies a 4xx (likely 400) when parsing fails (missing `dkim_record` key, malformed tag string, etc.). The exact error response shape is unspecified.

6. **Response payload shape**: Not specified. Likely `{"key_type": "RSA"}` or `{"key_type": "rsa"}` — the design phase should fix this before implementation.

7. **Test coverage scope**: Should the test cover (a) happy path only, (b) the missing-`k=`-defaults-to-RSA case, (c) a bad-payload 400 case? At minimum one passing test is required by the goal; the design phase should enumerate all required test cases.
