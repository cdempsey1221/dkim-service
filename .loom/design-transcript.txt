## Risks and open questions

1. **HTTP method**: The goal says "returns 200 if the json contains the dkim record" but doesn't specify `GET` vs `POST`. Given JSON body parsing, `POST` is the natural choice (consistent with `/split_by_value`), but this should be confirmed.

2. **`k=` tag absent → default RSA?** The canonical DKIM test record in this repo has no `k=` tag. RFC 6376 §3.3 defaults to `rsa` when `k=` is omitted. The new endpoint must decide: (a) default to `"rsa"` when absent, (b) return 400/error, or (c) document that it only works on records that explicitly carry `k=`.

3. **Case sensitivity of key type**: RFC 6376 says `k=rsa`; should the response normalise to uppercase `"RSA"` (as the task description implies) or echo the wire value `"rsa"`?

4. **Record format variability**: The existing record string is a flat whitespace-separated line (`name TXT <tags> ttl`). Real DKIM TXT records may have tags in different orders or combined differently (e.g., `v=DKIM1; k=rsa; p=...` as one token or spread). The parser must handle semicolon-separated tags within the body portion rather than relying on fixed token indices.

5. **What constitutes "parseable"?**: The task says return 200 "if … it can be parsed from the payload." This implies a 4xx (likely 400) when parsing fails (missing `dkim_record` key, malformed tag string, etc.). The exact error response shape is unspecified.

6. **Response payload shape**: Not specified. Likely `{"key_type": "RSA"}` or `{"key_type": "rsa"}` — the design phase should fix this before implementation.

7. **Test coverage scope**: Should the test cover (a) happy path only, (b) the missing-`k=`-defaults-to-RSA case, (c) a bad-payload 400 case? At minimum one passing test is required by the goal; the design phase should enumerate all required test cases.

# Design

## Approach

Add a `POST /dkim_key_type` endpoint to `dkim.py` backed by a new `parse_key_type(record)` helper. The helper scans the whitespace-tokenised record body for any token containing a `k=` tag (semicolon-separated), extracts its value, and uppercases it for the response. When the `k=` tag is absent, it defaults to `"RSA"` per RFC 6376 §3.3. Unparseable input (missing `dkim_record` key, non-string value) returns HTTP 400 with `{"error": "..."}`. The response on success is `{"key_type": "RSA"}`. A new test file `tests/test_dkim_key_type.py` covers the happy path (explicit `k=rsa`), the default-RSA path (no `k=` tag), and a 400 path (missing key in payload).

- **`tests/test_dkim_key_type.py`** — new test file with three test cases (explicit key type, absent key type defaults to RSA, missing payload key → 400)

## Step-by-step plan

1. **Add `parse_key_type` helper in `dkim.py`** (insert before the first `@app.route` decorator, after `split_key_by_val`):

   ```python
   def parse_key_type(record):
       for token in record.split():
           for tag in token.split(';'):
               tag = tag.strip()
               if tag.lower().startswith('k='):
                   return tag[2:].strip().upper()
       return 'RSA'  # RFC 6376 §3.3 default
   ```

   Logic: split the record on whitespace to get tokens, then split each token on `;` to get individual tag fragments, strip whitespace, find the one starting with `k=` (case-insensitive), and return its value uppercased. If no `k=` tag is found anywhere in the record, return `'RSA'`.

2. **Add `POST /dkim_key_type` route in `dkim.py`** (insert after the `split_by_value` route, before `healthz`):

   ```python
   @app.route('/dkim_key_type', methods=['POST'])
   def dkim_key_type():
       data = request.get_json()
       if not data or 'dkim_record' not in data:
           return jsonify({'error': 'missing dkim_record'}), 400
       key_type = parse_key_type(data['dkim_record'])
       return jsonify({'key_type': key_type}), 200
   ```

3. **Create `tests/test_dkim_key_type.py`** with three test cases (see Test plan below). Import pattern: `from dkim import app`.
